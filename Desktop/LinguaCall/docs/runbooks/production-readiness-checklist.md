# 실서비스 검증 및 안정화 체크리스트

핵심 기능 개발이 끝난 뒤, LinguaCall을 실제 출시 후보로 보기 전에 확인해야 할 항목들이다.

이 문서는 아래에 집중한다.

- 실서비스 검증
- 안정성 테스트
- 회귀 점검
- 운영자 대응 가능성

함께 볼 문서:

- [`launch-e2e-checklist.md`](./launch-e2e-checklist.md)
- [`supabase-phone-auth-manual.md`](./supabase-phone-auth-manual.md)
- [`toss-sandbox-manual.md`](./toss-sandbox-manual.md)

## 1. 배포 상태

아래가 모두 참이어야 한다.

- VPS 정상 실행
- DNS가 올바른 IP를 가리킴
- `web`, `api`, `worker`, `caddy` 실행 중
- HTTPS 정상 발급
- production env 파일 완성
- 최신 브랜치 커밋이 배포됨

명령:

```bash
docker compose --env-file infra/.env.production -f infra/docker-compose.yml ps
git log -1 --oneline
curl -i "https://API_DOMAIN/healthz"
curl -I "https://APP_DOMAIN"
```

통과 기준:

- 컨테이너 전체 실행 중
- `api`, `web` healthy
- 앱과 API가 HTTPS로 응답

## 2. 인증 검증

### 2.1 Supabase 테스트 OTP

- `/#/verify` 접속
- Supabase 테스트 번호로 OTP 요청
- OTP 입력
- `/#/session` 이동 확인
- 새로고침 후 로그인 유지 확인
- 로그아웃 확인
- 보호 경로 재진입 시 로그인 화면으로 복귀 확인

### 2.2 실제 SMS OTP

Twilio가 Supabase에 연결된 뒤 진행한다.

- 실제 기기 번호로 OTP 요청
- SMS 도착 확인
- 한 번은 정상 인증
- 잘못된 코드 거부 확인
- 만료된 코드 거부 확인
- 재전송 UX 확인

통과 기준:

- 로그인 성공
- 세션 복구 성공
- 로그아웃 성공
- 실제 SMS 도착 및 인증 성공

## 3. 결제 검증

### 3.1 샌드박스

- `/#/billing` 접속
- 현재 플랜 상태 확인
- Toss 샌드박스 결제 시작
- 결제 완료
- 앱 복귀
- 구독 상태 업데이트 확인
- 새로고침 후 상태 유지 확인

### 3.2 실패 시나리오

- 결제 취소
- UI가 정상 복구되는지 확인
- 중단되거나 잘못된 결제 경로 시도
- 깨진 구독 상태가 생기지 않는지 확인

통과 기준:

- checkout 시작 성공
- confirm 성공
- 상태가 새로고침 후에도 유지
- 실패 결제가 상태를 망치지 않음

## 4. 세션 생명주기 검증

로그인된 일반 사용자 기준으로 확인한다.

- 세션 생성
- 목록 반영 확인
- 상세 진입 확인
- 새로고침 후 유지 확인
- 지원하면 수정/재예약 확인
- 지원하면 취소 확인

통과 기준:

- 목록과 상세가 일치
- 고스트 세션이나 중복 상태 없음

## 5. 라이브 음성 검증

### 5.1 기본 통화

- 마이크 허용
- 라이브 세션 시작
- 연결 성공 확인
- 짧게 발화
- 모델 응답 확인
- 세션 종료

### 5.2 안정성 점검

- 연속 시작/종료 반복
- 느린 연결 또는 늦은 마이크 허용 테스트
- `connecting` 상태에서 종료 테스트
- AI 발화 후 종료 테스트
- 통화 중 또는 직후 새로고침 테스트

통과 기준:

- 시작 성공
- 종료 안정적
- `connecting`에 고정되지 않음
- 런타임 오류가 치명적으로 남지 않음

## 6. 리포트 파이프라인 검증

- 세션 종료
- worker 처리 대기
- 리포트 열기
- 점수, 요약, 교정, 다음 액션 렌더링 확인
- 리포트 새로고침 확인

통과 기준:

- 수동 개입 없이 리포트 생성
- UI가 깨지지 않음
- 내용 누락이나 명백한 손상 없음

## 7. 브라우저 및 기기 교차 검증

최소 기준:

- 데스크톱 Chrome
- 모바일 Safari 또는 모바일 Chrome

확인 항목:

- 로그인
- 결제
- 세션 생성
- 라이브 통화 시작
- 라이브 통화 종료
- 리포트 화면

통과 기준:

- 치명적 레이아웃 문제 없음
- 모바일에서 상호작용 불가 상태 없음
- 마이크 권한 흐름이 이해 가능

## 8. UI 신뢰감 점검

처음 보는 사용자처럼 본다.

- 랜딩이 신뢰감 있게 보이는가
- 인증 화면이 안전하고 이해 가능한가
- 세션 화면에서 다음 행동이 명확한가
- 결제 화면이 설득력 있는가
- 리포트 화면이 실제 가치 있어 보이는가
- 개인정보/이용약관이 읽을 수 있는 상태인가
- 깨진 문자열이 없는가
- placeholder 문구가 없는가
- dead CTA가 없는가

통과 기준:

- 모르는 사람이 봐도 배포 가능한 제품처럼 보임
- 신뢰를 깎는 UI 문제가 남아 있지 않음

## 9. 에러 및 복구 시나리오

가능하면 아래를 직접 유도하거나 시뮬레이션한다.

- 잘못된 OTP
- 만료된 OTP
- 결제 취소
- 마이크 없음
- 마이크 권한 거부
- 연결 중단
- 리포트 지연
- 인증 세션 만료

확인 항목:

- UI가 이해 가능한 에러 문구를 보여주는가
- 앱이 unusable 상태에 빠지지 않는가
- 재시도 경로가 분명한가

통과 기준:

- 흔한 실패마다 복구 경로가 있다

## 10. 로그와 운영자 가시성

운영자가 장애를 빠르게 파악할 수 있어야 한다.

확인 항목:

- `api` 로그가 요청 실패를 읽기 쉽게 남기는가
- `worker` 로그가 처리 실패를 읽기 쉽게 남기는가
- 민감한 토큰/시크릿이 로그에 찍히지 않는가
- health check가 유효한가
- manual worker trigger가 동작하는가

명령:

```bash
docker compose --env-file infra/.env.production -f infra/docker-compose.yml logs --tail=200 api
docker compose --env-file infra/.env.production -f infra/docker-compose.yml logs --tail=200 worker
curl -i -X POST "https://API_DOMAIN/workers/run" -H "x-worker-token: YOUR_WORKER_SHARED_SECRET"
```

통과 기준:

- 장애 원인을 추측이 아니라 로그로 파악 가능

## 11. 데이터 무결성 점검

실제 플로우를 돌린 뒤 DB를 spot check 한다.

확인 항목:

- 실제 인증 사용자당 user row 하나
- 결제 상태가 실제 결제 흐름과 일치
- 세션 상태 전이가 정상
- 리포트 row가 한 번만 생성
- 중복, orphan 데이터가 명백히 보이지 않음

통과 기준:

- 인증, 결제, 세션, 리포트 상태가 내부적으로 일관됨

## 12. 출시 판단

### 라이브 베타 가능

아래가 모두 맞으면 제한적 실사용자 투입 가능으로 본다.

- 배포 상태 정상
- 실제 OTP 정상
- 결제 정상
- 세션 생성 정상
- 라이브 통화 정상
- 리포트 정상
- 복구 시나리오 수용 가능
- 치명적 신뢰 저하 UI 없음

### 아직 출시 불가

아래 중 하나라도 남아 있으면 launch-ready로 보지 않는다.

- 실제 SMS 도착이 불안정
- 결제 confirm 불안정
- 라이브 세션 시작 또는 종료가 자주 깨짐
- 리포트가 장시간 pending
- 로그아웃 또는 세션 복구가 깨짐
- 모바일 경험이 데스크톱보다 의미 있게 나쁨
- 운영자가 로그만 보고 장애를 이해할 수 없음

## 13. 최종 사인오프 기록

출시 전 아래를 명시적으로 기록한다.

- 테스트 날짜
- 배포된 커밋 SHA
- 테스트한 사람
- 테스트한 환경
- 테스트한 브라우저
- 테스트한 기기
- 남은 known issue
- 최종 판단: `go`, `go with known issues`, `no-go`
