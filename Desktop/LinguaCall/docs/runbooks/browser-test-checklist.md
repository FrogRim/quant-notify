# Browser Test Checklist

## 목적

이 문서는 프론트 리디자인 이후 실제 브라우저에서 확인해야 할 핵심 사용자 플로우를 정리한다. 목표는 `결제 전환율 상승`에 직접 영향을 주는 신뢰감, 흐름, CTA 노출, 오류 복구를 눈으로 검증하는 것이다.

## 테스트 환경

- 데스크톱 브라우저 1종
  - Chrome 권장
- 모바일 브라우저 1종
  - iPhone Safari 또는 Android Chrome 중 최소 1개
- 테스트 계정
  - OTP 로그인 가능한 전화번호
- 테스트 결제
  - Toss sandbox 키 연결 상태

## 1. 첫 화면 인상

경로:
- `/`

체크:
- 로고/서비스명이 허술하지 않게 보인다.
- `짧고 가볍게 꾸준히 연습` 메시지가 첫 화면에서 이해된다.
- 메인 CTA가 바로 보인다.
- 모바일에서 히어로 영역이 너무 길지 않다.
- 하단 정책 링크가 화면을 가리지 않는다.

성공 기준:
- 10초 안에 서비스 목적과 다음 행동이 이해된다.

## 2. OTP 로그인

경로:
- `/verify`

체크:
- 전화번호 입력 필드가 자연스럽다.
- 인증번호 요청 버튼이 눈에 잘 띈다.
- 요청 후 상태 메시지가 자연스럽다.
- 인증번호 입력 단계 전환이 명확하다.
- 잘못된 번호/코드 입력 시 오류가 읽기 쉽다.

성공 기준:
- 처음 보는 사용자도 막힘 없이 인증 흐름을 따라갈 수 있다.

## 3. 로그인 유지

체크:
- OTP 인증 후 `/session`으로 정상 이동한다.
- 브라우저 새로고침 후에도 다시 OTP를 요구하지 않는다.
- 새 탭으로 다시 열어도 세션이 유지된다.
- 로그아웃 후에는 보호 페이지 접근 시 다시 로그인으로 이동한다.

성공 기준:
- 반복 방문 UX가 앱처럼 자연스럽다.

## 4. 세션 홈

경로:
- `/session`

체크:
- 첫 화면에서 `지금 시작` CTA가 가장 눈에 띈다.
- 예약, 최근 리포트, 현재 플랜 정보가 한 화면에 과밀하지 않다.
- 세션 생성 카드가 허브를 압도하지 않는다.
- 모바일에서 카드 간 간격과 버튼 폭이 자연스럽다.

성공 기준:
- 사용자가 망설이지 않고 다음 행동을 고를 수 있다.

## 5. 세션 생성

체크:
- 언어, 레벨, 주제, 길이 선택이 자연스럽다.
- 일본어/프랑스어 포함 각 언어 선택이 정상 동작한다.
- 즉시 세션 생성 후 목록과 상태가 자연스럽게 갱신된다.
- 예약 생성도 시각적으로 구분된다.

성공 기준:
- 세션 생성 자체가 복잡해 보이지 않는다.

## 6. 실시간 통화

체크:
- 마이크 권한 요청이 자연스럽다.
- 연결 중 상태 문구가 이해 가능하다.
- 연결 후 transcript가 무리 없이 쌓인다.
- 종료 버튼이 명확하다.
- 종료 후 즉시 홈/상세 상태가 정상 반영된다.

성공 기준:
- 통화 시작과 종료가 불안정해 보이지 않는다.

## 7. 결제 페이지

경로:
- `/billing`

체크:
- 현재 플랜과 업그레이드 선택지가 한눈에 구분된다.
- 각 플랜의 대상/차이가 읽기 쉽다.
- 가격, 포함 분수, 세션 길이가 빠르게 이해된다.
- 추천 플랜 강조가 과하지 않다.
- 무료 사용자도 압박감보다 자연스러운 업그레이드 제안으로 느낀다.

성공 기준:
- 단순 가격표가 아니라 설득형 제안처럼 보인다.

## 8. Toss 결제

체크:
- 결제 버튼 클릭 후 Toss sandbox 흐름이 열린다.
- 성공 후 `/billing` 복귀와 성공 배너가 자연스럽다.
- 현재 플랜 상태가 실제로 갱신된다.
- 실패/취소 시 복귀 문구가 읽기 쉽다.

성공 기준:
- 결제 전환 흐름이 끊기지 않는다.

## 9. 리포트 화면

경로:
- `/report/:reportId`

체크:
- 총점/요약/추천 액션이 상단에서 바로 보인다.
- 세부 분석은 읽고 싶은 만큼 내려가며 볼 수 있다.
- 카드, 배지, 여백이 세션/결제 화면과 같은 제품처럼 보인다.
- 모바일에서 점수/카드가 눌리거나 잘리지 않는다.

성공 기준:
- 리포트가 “점수 던져주는 화면”이 아니라 학습 피드백처럼 느껴진다.

## 10. 정책 페이지

경로:
- `/privacy`
- `/terms`

체크:
- 인코딩 깨짐이 없다.
- 문서 레이아웃이 깔끔하다.
- 제목, 소제목, 본문 가독성이 충분하다.

성공 기준:
- 정책 페이지가 제품 신뢰를 깎지 않는다.

## 11. 반응형 최종 확인

데스크톱:
- 카드 폭이 과하게 넓지 않다.
- 빈 여백이 지나치지 않다.

모바일:
- CTA가 첫 화면에 남는다.
- 버튼이 너무 작지 않다.
- 가로 스크롤이 발생하지 않는다.
- 푸터나 배너가 입력 영역을 가리지 않는다.

## 완료 기준

아래를 만족하면 브라우저 QA 통과로 본다.

- 로그인, 세션, 결제, 리포트, 정책 페이지가 모두 정상 표시된다.
- OTP 재로그인 강제가 사라진다.
- 세션 생성/통화 시작/통화 종료가 정상 동작한다.
- Toss sandbox 결제가 정상 동작한다.
- 모바일과 데스크톱 모두 치명적 레이아웃 문제 없음
